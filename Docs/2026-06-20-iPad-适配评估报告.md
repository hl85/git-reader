# GitReader iPad 适配评估报告

> 评估日期：2026-06-20
> 评估范围：`GitReader/` 全部 Swift 源码 + `project.yml` 项目配置
> 评估目标：当前应用在 iPad 上的适配完成度与体验质量

---

## 总体结论

**适配完成度：约 25%**

项目配置层面已就绪，但 UI 层面的 iPad 适配几乎为零。当前应用在 iPad 上的体验本质上是「放大版 iPhone」，未利用大屏任何优势。

---

## ✅ 已就绪：项目配置

`project.yml` 第 86 行 `TARGETED_DEVICE_FAMILY: "1,2"` 声明支持 iPhone + iPad；第 77-81 行配置了 iPad 四个方向（Portrait / PortraitUpsideDown / LandscapeLeft / LandscapeRight）；部署目标 iOS 17，可使用最新 `NavigationSplitView` API。这些是基础前提，已正确配置。

---

## ❌ 严重问题 1：导航架构 — 零 `NavigationSplitView`

这是最核心的缺陷。全代码库**完全没有** `NavigationSplitView`，也没有任何 `horizontalSizeClass` 判断。

- `GitReader/Views/FileList/FileListView.swift:30` 以 `NavigationStack` 为根
- `GitReader/Views/FileList/FileListView.swift:159-163` 通过 `.navigationDestination` 把 `NoteReaderView` push 进栈

```30:30:GitReader/Views/FileList/FileListView.swift
        NavigationStack {
```

```159:163:GitReader/Views/FileList/FileListView.swift
            .navigationDestination(for: FileItem.self) { file in
                NoteReaderView(
                    fileURL: file.url,
                    noteIndex: noteIndex
                )
```

**影响**：在 iPad 上，`NavigationStack` 会把详情页**全屏推入覆盖**整个屏幕。结果是：iPad 12.9 寸横屏时，打开一篇笔记后整个屏幕只显示这一篇笔记的正文（左右大量留白或文字铺满），文件列表被完全隐藏，必须点返回才能切换笔记。这彻底浪费了 iPad 大屏并排浏览的核心优势。

**标准做法**：iPad 应使用 `NavigationSplitView`，左侧侧边栏常驻文件列表，右侧显示阅读详情，点击笔记即更新右侧而非覆盖全屏。iOS 16+ 的 `NavigationSplitView` 在 iPhone 上自动退化为 push 行为，一套代码两端通吃。

---

## ❌ 严重问题 2：阅读内容无宽度限制

`GitReader/Theme/ClaudeTypography.swift:27` `readerHorizontalPadding: 20` 是固定常量，`GitReader/Views/NoteReader/NoteReaderView.swift:132` 直接套用：

```27:27:GitReader/Theme/ClaudeTypography.swift
    static let readerHorizontalPadding: CGFloat = 20
```

```132:132:GitReader/Views/NoteReader/NoteReaderView.swift
        .padding(ClaudeTypography.readerHorizontalPadding)
```

**影响**：iPad 横屏正文区域宽度可达 ~1000pt，减去 40pt padding 后仍有 ~960pt。一段正文会横跨整个屏幕宽度，单行字符数远超 80 字符的可读性上限，读者需要大幅转头/移眼，阅读体验急剧下降。阅读类应用（如 Apple Books、Bear、Obsidian）在 iPad 上都会将正文宽度限制在约 600-720pt 并居中。

---

## ❌ 严重问题 3：完全无尺寸类（Size Class）适配

全代码库**零** `@Environment(\.horizontalSizeClass)`、**零** `userInterfaceIdiom == .pad` 判断。所有布局在 iPhone 和 iPad 上完全相同。唯一的 `UIDevice` 使用是 `GitReader/Views/NoteReader/NoteReaderView.swift:1222-1233` 用于检测屏幕方向（导出截图逻辑），与布局适配无关。

这意味着没有任何地方根据「是不是宽屏 / 是不是 iPad」做差异化：

- 列表没有从单列变多列
- 卡片网格没有从 1 列变 2-3 列
- 工具栏没有为大屏增加快捷操作

---

## ⚠️ 中等问题：布局细节

### 1. 悬浮菜单硬编码定位

`GitReader/Views/FileList/FileListView.swift:117` `.padding(.top, 106)` 依赖搜索栏+过滤器精确叠加高度。一旦 iPad 上布局/字号变化，下拉菜单位置会错位。

### 2. Toolbar placement

`GitReader/Views/FileList/FileListView.swift:173`、`GitReader/Views/NoteReader/NoteReaderView.swift:163` 均用 `.navigationBarTrailing`。在 iPad 分栏布局下，应改用 `.topBarTrailing`，并考虑侧边栏底部放同步/设置按钮。

### 3. Sheet 未指定 detents

`GitReader/Views/NoteReader/NoteReaderView.swift:245` 的 `SetPropertiesSheet` 在 iPad 上默认居中 form sheet，但未显式控制尺寸，不同 iPad 尺寸下表现不一致。

### 4. `.navigationBarTitleDisplayMode(.inline)` 全局使用

在 iPad 详情页，大标题（`.large`）通常更合适；至少在 split view 的详情列应允许大标题。

---

## ✅ 正面基础

- 纯 SwiftUI 声明式，多用 `VStack` / `HStack` / `Spacer` / `.frame(maxWidth: .infinity)`，具备基本弹性
- 已采用 iOS 16+ 的 `NavigationStack`，迁移到 `NavigationSplitView` 是平滑升级
- 文件列表是树形结构，天然适合做 split view 侧边栏

---

## 📋 优先级建议

| 优先级 | 改造项 | 预期收益 |
|---|---|---|
| P0 | 将 `FileListView` 的 `NavigationStack` 重构为 `NavigationSplitView`（侧边栏=文件树，详情=`NoteReaderView`） | iPad 体验质变，并排浏览 |
| P0 | 阅读内容加 `.frame(maxWidth: 720)` 居中限制 | 解决长行可读性 |
| P1 | 引入 `@Environment(\.horizontalSizeClass)`，在 `.regular` 下做差异化布局 | 多列网格/更宽卡片 |
| P1 | 设置页 / `RepoConfigView` 在 iPad 用 `NavigationSplitView` 或 `form` sheet detents | 表单体验 |
| P2 | Toolbar placement 改 `.topBarTrailing`，悬浮菜单改用 `geometry`/锚点定位 | 细节稳健性 |

---

## 总结

项目「能跑在 iPad 上」，但谈不上「适配 iPad」。核心瓶颈在于导航架构（单栈 push）和内容宽度无限制——这两项是阅读类应用在 iPad 上体验的决定性因素，建议作为首要改造目标。
